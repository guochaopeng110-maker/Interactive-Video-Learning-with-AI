## Problem Statement

The current UI prototype direction is not yet structured as a complete product experience.

The existing prototype work is centered on an early teacher-first sequence, which is useful for exploring `Material` upload and authoring interactions, but it does not yet define a stable end-to-end product structure across teacher, student, and admin roles. Because of that, the team lacks a shared blueprint for:

- how the three role workspaces relate to the same `Course`, `KnowledgeGraph`, and `Lesson`
- which pages are shared versus role-specific
- where permission boundaries must be visible in the UI
- how prototype routes should be grouped and sequenced for later implementation

Without a role-based PRD, the team risks continuing to add prototype screens as isolated pages, producing a fragmented navigation model and making later implementation planning less reliable.

## Solution

Redesign the complete UI prototype as a role-based product architecture built on the shared course domain model.

Instead of extending the old linear 9-screen order, the prototype will be reorganized into four layers:

- a shared entry layer for authentication, role selection, and profile/help surfaces
- a teacher workspace for `Course` authoring, `Material` review, `KnowledgeGraph` maintenance, `Lesson` composition, preview, publish, and analytics
- a student workspace for course entry, knowledge-graph-guided learning, lesson playback, remediation, and progress tracking
- an admin workspace for `Major` and `Course` governance, user and role management, publication oversight, and organization analytics

This redesign keeps the domain model stable while making the role differences explicit in route structure, page responsibilities, and action boundaries. It gives later prototype work a coherent skeleton and gives downstream issue planning a shared language for slicing implementation.

## User Stories

1. As a teacher, I want a dedicated workspace for preparing and improving course content, so that authoring tasks are not mixed with student or admin concerns.
2. As a teacher, I want to enter a single `Course` workspace and see its `Material`, `KnowledgeGraph`, and `Lesson` together, so that I can manage one teaching context at a time.
3. As a teacher, I want to upload `Material` and review AI extraction results in one place, so that I can confirm `KnowledgePoint` efficiently.
4. As a teacher, I want low-confidence AI suggestions to be surfaced first, so that I can focus review effort where it matters most.
5. As a teacher, I want an editable `KnowledgeGraph`, so that I can shape the course structure before lesson composition.
6. As a teacher, I want to compose `Lesson` flows from confirmed `KnowledgePoint`, so that authoring follows the shared course model.
7. As a teacher, I want to preview a `Lesson` in a student-like mode before publishing, so that I can validate flow and clarity.
8. As a teacher, I want to see learning analytics tied back to `LessonNode` and `KnowledgePoint`, so that I can iterate on the course design.
9. As a student, I want a dedicated learning workspace, so that the platform feels centered on what I need to study next.
10. As a student, I want to see my `Course` list and progress clearly, so that I can resume learning quickly.
11. As a student, I want a read-only `KnowledgeGraph` view of the course, so that I can understand what I have learned and what is still ahead.
12. As a student, I want to enter a published `Lesson` directly from the graph or course page, so that navigation feels natural.
13. As a student, I want the lesson player to combine video, questions, and branch feedback, so that learning stays focused in one place.
14. As a student, I want remediation to be explicit in the experience, so that mistakes lead to structured recovery instead of confusion.
15. As a student, I want `GroundedAssistant` answers to stay aligned with course `Material`, so that help inside the lesson remains trustworthy.
16. As a student, I want a progress page focused on mastery and next actions, so that I know how to continue effectively.
17. As an admin, I want a separate governance workspace, so that organizational operations do not get mixed into teacher authoring screens.
18. As an admin, I want to manage `Major` and `Course` structure, so that the academic catalog stays consistent.
19. As an admin, I want to manage users and role assignments, so that workspace access reflects organizational responsibility.
20. As an admin, I want to inspect publication state and audit concerns, so that governance decisions happen without directly editing teaching content.
21. As an admin, I want organization-level analytics, so that I can monitor adoption and operational health across courses.
22. As a product designer, I want shared entry pages for login, role selection, and profile, so that the product shell is consistent across all roles.
23. As a product designer, I want shared `Course`, `KnowledgeGraph`, and `Lesson` visual language across workspaces, so that the product feels coherent while actions still respect permissions.
24. As a frontend engineer, I want prototype routes grouped by role, so that implementation can grow from a clear folder and navigation structure.
25. As a frontend engineer, I want each screen to have explicit goals, inputs, outputs, and next steps, so that page implementation does not depend on implicit assumptions.
26. As a frontend engineer, I want permission boundaries expressed in the prototype structure, so that students never inherit draft or authoring controls by accident.
27. As a product owner, I want the prototype review order to follow shared entry, teacher, student, then admin workspaces, so that stakeholders can understand the product model progressively.
28. As a product owner, I want the implementation order to start with shared shell and teacher authoring before student and admin workspaces, so that downstream screens are built on stable domain structure.
29. As an implementation planner, I want reusable components to be identified across workspaces, so that prototype construction does not duplicate the same UI concepts three times.
30. As an implementation planner, I want the role-based prototype redesign captured as a standalone PRD, so that later issue breakdown can target this work specifically rather than the broader platform scope.

## Implementation Decisions

- The prototype redesign is scoped as a role-based UI architecture initiative, not a change to the underlying domain model.
- The existing teacher-first prototype seed is treated as a partial starting point rather than the final navigation model.
- The product model continues to center on shared domain objects: `Major`, `Course`, `Material`, `KnowledgePoint`, `KnowledgeGraph`, `Lesson`, and `LessonNode`.
- The prototype is reorganized into four route groups:
  - shared entry layer
  - teacher workspace
  - student workspace
  - admin workspace
- Shared entry responsibilities include authentication, role entry, role switching, profile, notification, and help surfaces.
- Teacher workspace responsibilities include `Course` context, `Material` ingestion, AI review, `KnowledgeGraph` maintenance, `Lesson` authoring, preview, publish, and analytics.
- Student workspace responsibilities include course access, knowledge-graph-guided navigation, lesson playback, remediation, and progress reflection.
- Admin workspace responsibilities include academic catalog structure, user and role management, governance flows, and organization analytics.
- The same `Course` model must appear across all roles, but the available actions differ by permission.
- The same `KnowledgeGraph` visual language should be preserved across teacher and student views, while teacher behavior is editable and student behavior is read-only.
- The same `Lesson` source should be reused across teacher preview and student playback, while interaction affordances differ by mode.
- `Material-Constrained Generation` remains a binding product rule and must be reflected in teacher-facing review and student-facing assistant trust cues.
- The prototype must make final approval boundaries explicit: teacher review gates publishable AI content, students never see draft states, and admins govern access and publication state without entering authoring flows.
- The navigation contracts for the redesign are:
  - shared entry routes lead into role homes
  - teacher publication makes `Lesson` visible to student routes
  - admin governance affects access, visibility, ownership, and audit state
- The major modules to build or modify are:
  - `Role Workspace Shell`
  - `Teacher Authoring Workspace`
  - `Student Learning Workspace`
  - `Admin Governance Workspace`
  - `Shared Course View Model`
  - `Prototype Navigation Contracts`
- Deep-module opportunities for the prototype layer:
  - `Role Workspace Shell` should encapsulate role routing, shared chrome, and role-aware navigation in one stable interface.
  - `Shared Course View Model` should encapsulate how one course entity is transformed into teacher, student, and admin presentations without duplicating business meaning.
  - `Prototype Navigation Contracts` should encapsulate route-to-route assumptions so visibility and permission boundaries are enforced consistently.
- The redesign stays aligned with existing architecture decisions:
  - modular monolith remains the implementation target
  - REST plus shared DTOs remains the application integration model
  - semi-automatic knowledge extraction remains the teacher review assumption
  - unified vector-backed storage remains unaffected by this UI scope

## Testing Decisions

- Good tests should verify external behavior and permission outcomes rather than implementation details such as internal component structure.
- Priority modules to test:
  - `Role Workspace Shell`
  - `Shared Course View Model`
  - `Teacher Authoring Workspace`
  - `Student Learning Workspace`
  - `Prototype Navigation Contracts`
- `Role Workspace Shell` tests should verify role-based entry, correct landing behavior, and consistent shared shell behavior.
- `Shared Course View Model` tests should verify that identical domain entities are rendered with different allowed actions for teacher, student, and admin contexts without semantic drift.
- `Teacher Authoring Workspace` tests should verify the screen-to-screen progression from `Material` review through `KnowledgeGraph`, `Lesson` authoring, preview, and publish.
- `Student Learning Workspace` tests should verify the screen-to-screen progression from `Course` entry through `KnowledgeGraph`, `Lesson`, remediation, and progress.
- `Prototype Navigation Contracts` tests should verify that unpublished authoring content does not appear in student routes and that admin routes do not expose teacher authoring actions.
- A good test in this area should assert user-visible navigation, page states, and permission boundaries, not incidental layout composition.
- Prior art in the repository is still light, so this work should establish reference patterns for route-group tests, role-aware view-model tests, and prototype interaction tests that later issues can copy.

## Out of Scope

- production backend API changes
- persistence schema changes
- content-generation algorithm changes
- retrieval or vector storage changes
- full visual polish for every screen
- final copywriting for all prototype states
- implementation issue breakdown
- production-ready access control enforcement outside the prototype scope

## Further Notes

- This PRD is intentionally narrower than the overall platform PRD. It focuses on the role-based UI prototype redesign needed before prototype implementation continues.
- The redesign should be treated as the structural parent for later route-level prototype tasks.
- The resulting issue breakdown should preserve the role narrative: shared entry first, then teacher, then student, then admin.
- The prototype should continue using the project glossary from `CONTEXT.md` so future design and engineering work stays aligned.
