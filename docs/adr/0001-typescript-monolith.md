# TypeScript 全栈 + 模块化单体

构建一个面向护理教育的 AI 交互式视频学习平台，MVP 范围为 1 学校、2 教师、100 学生、4 周试运行。需要选择技术栈和架构形态。

## 决策

采用 **TypeScript 全栈（Next.js + NestJS）+ 模块化单体**，PRD 定义的四个 Engine 先作为模块内边界，不拆分为独立服务。

## 拒绝的选项

- **Python 后端（FastAPI/Flask）**：AI 生态更强，但前后端分裂为两种语言，MVP 阶段团队需要维护两套类型系统、两套工具链，收益不足以抵消成本。
- **微服务架构**：100 名学生规模下不需要独立部署、独立扩缩容。过早拆分引入网络延迟、分布式事务、运维复杂度，且四个 Engine 之间的交互在进程内更简单可靠。

## 后果

- 四个 Engine 的接口在 NestJS 模块间要维持清晰边界，为未来可能的拆分做准备。
- 单体部署简化 CI/CD，一个 Docker 镜像覆盖全栈。
